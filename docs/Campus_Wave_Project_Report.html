Executive Summary

Campus Wave is a Flutter app for exploring LGS campuses, facilities, staff, and managing professor appointments. It integrates Firebase Auth for sign-in, Provider for state, GoRouter for navigation, SharedPreferences for local bookings, and a clean emerald-themed UI. Users browse campuses, view per-campus facilities galleries, find professors, book appointments, search across campus data, and access a basic AI chatbot screen.

App Overview

  • Platforms: Flutter/Dart, Android emulator support.
  • Key libs: firebase_core, firebase_auth, provider, go_router, shared_preferences, image_picker, intl, flutter_localizations.
  • Assets: campus covers in assets/Lgs Picscampus/; facilities gallery in assets/Lgsgallerypics/.
  • Routing: centralized in lib/router.dart with auth-aware redirects.

Architecture & Dependencies

Entry: lib/main.dart initializes Firebase and Providers, builds MaterialApp.router with GoRouter.

Routing: lib/router.dart defines root/shell navigator keys, named routes, and auth redirects.

Data: Enriched campus data (lib/data/campuses.dart), per-campus professors (lib/data/campus_professors.dart), appointments sample (lib/data/appointments.dart), search index (lib/data/search_repository.dart).

Navigation & Routing

Routes include login, signup, home, campus info/detail, facilities, professors (global and campus-specific), professor detail, appointments, search, cafeteria, library, chatbot, event/news details. Auth redirects gate sensitive routes; Home is the central hub after login.

Screens & Features

  • lib/homescreen.dart: Home hub with quick access (Campus Info, Facilities, AI Chatbot), Student Services (Professors, Appointments), search.
  • lib/screens/campus_info_screen.dart: Campus list + tabs (Overview, Facilities, Staff, History). Overview decluttered — removed Book Appointment / View Library / Message Office buttons.
  • lib/screens/facilities_screen.dart: Per-campus cover mapping and gallery (PageView), breadcrumbs, search integration.
  • lib/screens/professors_screen.dart: Global grouped view by campus: bio, skills chips, achievements, available slots, booking.
  • lib/screens/campus_professors_screen.dart: Campus-specific directory; tap to detail/booking.
  • lib/screens/professor_detail_screen.dart: Booking flow with login prompt, confirmation, local persistence, auto-nav to Appointments.
  • lib/screens/appointments_screen.dart: Per-user list, cancel/clear-all, login prompt.
  • lib/screens/search_screen.dart + lib/data/search_repository.dart: Index build, fuzzy search, slug navigation.
  • lib/screens/user_profile_screen.dart: Display name and photo management; avatar shows initials/photo.
  • lib/screens/library_screen.dart, lib/screens/cafeteria_screen.dart: Placeholder galleries and copy.
  • lib/screens/chatbot_screen.dart: AI Chatbot screen present.

Data Models

  • Campus (lib/data/campuses.dart): name, location, imageUrl, description, features, history, academicHighlights, facilityHighlights, photoCaptions.
  • ProfessorExtended (lib/data/campus_professors.dart): extends Professor with skills and achievements; plus availableSlots.
  • Appointment (lib/data/appointments.dart): id, userId, professorId, start, end, location.

Authentication

Firebase Auth supports email/password and Google Sign-In. Post-auth routes call context.go('/home'). Login and Signup implemented in lib/loginscreen_new.dart and lib/signup.dart.

State & Persistence

  • ThemeProvider (lib/theme/theme_provider.dart): dark/light themes persisted in SharedPreferences.
  • Appointments: stored locally in SharedPreferences (bookings JSON), filtered by current Firebase user.
  • User Profile: photo saved as base64 in SharedPreferences; display name via Firebase user.

Local Storage Schema

Bookings are stored as a JSON list under the key bookings:

[
  {
    "professorId": "p1",
    "professorName": "Dr. Ayesha Khan",
    "slot": "Tue 11:00",
    "userUid": "<firebase uid>",
    "createdAt": "2025-11-30T12:34:56.000Z"
  }
]

Profile photo is stored as base64 under profilePhotoBase64. Theme preference under isDarkMode.

Search

Search index built from campus data in lib/data/search_repository.dart with simple fuzzy scoring and slug navigation via lib/screens/search_screen.dart.

Index & Scoring

  • Index items: campuses (name/location), facility highlights, academic highlights.
  • Slug generation matches router (campusSlug()).
  • Fuzzy scoring prioritizes starts-with matches, then contains; results sorted by score descending.

Assets Strategy

  • Campus covers: assets/Lgs Picscampus/*.png mapped by campus name.
  • Facilities gallery: assets/Lgsgallerypics/ with explicit per-campus filenames.
  • LgsImage (lib/widgets/lgs_image.dart) prefers asset, falls back to network, then placeholder.

Theming & Localization

  • Emerald-themed dark/light palettes; ThemeToggleButton supports switching.
  • Localized strings via lib/l10n/app_localizations.dart.

Build & Run

Firebase initialized in lib/main.dart. Ensure flutter pub get completes and emulator is ready.

flutter pub get
flutter run -d emulator-5554

Note: Emulator disconnections may occur; re-run if needed.

Auth Redirects: GoRouter sends unauthenticated users to /login and redirects authenticated users away from login/signup to /home.

Known Issues

  • Asset path casing must match exact filenames.
  • Potential go_router duplicate page-key assertions in edge cases; unique keys or redirect timing may help.
  • Library/Cafeteria screens are placeholders.
  • Appointments are local-only (no backend sync).
  • Intermittent emulator connection loss during flutter run; retry recommended.

Future Enhancements

  • Persist bookings/profile in Firestore.
  • Slot conflict detection and calendar integration.
  • Expand chatbot capabilities and campus resources.
  • Index professors in Search and add filters.
  • Add tests, CI, error tracking; improve accessibility and performance.
  • Replace local JSON schema with normalized collections in Firestore (users, professors, appointments).

16) User Personas & Key Use Cases

Personas

  • Prospective Student: explores campuses, facilities, professors, and books guidance sessions.
  • Current Student: searches facilities, books appointments, checks staff info, manages profile.
  • Parent/Guardian: reviews campus info, facilities highlights, staff profiles, and contacts office.
  • Staff/Professor: reviews own profile, availability slots, and appointments list.

Use Cases

  • Browse campus list → open campus detail tabs → view history and facilities.
  • Search for "library" or "sports" → navigate to campus or facilities relevant pages.
  • Open professors directory → filter by campus → select professor → book slot → view appointment.
  • Set profile photo and display name → avatar updates across app.

17) Detailed Functional Specification

Home

  • Displays quick access cards: Campus Info, Facilities, AI Chatbot; Student Services: Professors, Appointments.
  • Search via AppBar or actions opens /search.
  • ReadableWidth constrains content on wide layouts; consistent section headers.

Campus Info

  • Campus list with cover image (asset-first via LgsImage) and teaser description.
  • Campus Detail has tabs: Overview (badges, description), Facilities (highlights grid), Staff (opens campus professors), History (expand/collapse).
  • Overview intentionally decluttered: removed Book Appointment, View Library, and Message Office actions.

Facilities

  • Campus cover image mapping via assetCoverForCampus.
  • Gallery per campus (PageView) sourced from assets/Lgsgallerypics/.
  • Breadcrumbs and search access; graceful image fallback behavior.

Professors

  • Global grouped view by campus, each with professor card.
  • Card shows name, title, department, bio, skills chips, achievements list, available slots, and booking action.
  • Campus-specific list available from Staff tab.

Professor Detail

  • Shows extended info (skills/achievements), office, bio, and slots.
  • Booking: prompts login if needed → confirmation dialog → SharedPreferences persist → auto-nav to appointments.
  • Error states: failed persistence, missing professor → fallback UI.

Appointments

  • Lists only current user’s bookings; supports cancel (remove item) and clear-all.
  • Prompts to login if unauthenticated.
  • Swipe-to-dismiss or explicit actions for cancellation.

Search

  • Index built from campuses, facility and academic highlights; suggestion tags (e.g., library, sports).
  • Results sorted via simple fuzzy scoring; navigates to campus routes.

Profile & Avatar

  • Set display name and photo (base64 in SharedPreferences); avatar shows initials or stored photo.
  • UserAvatarButton reads stored photo and Firebase user initials.

Library & Cafeteria

  • Present galleries and descriptive copy; future integration with campus-specific assets.

18) Routing Map & Navigation Flows

Hierarchical Map

  • /login, /signup, /createAccount
  • Shell: /home, /campusInfo, /facilities, /appointments, /professors, /search, /cafeteria, /library, /chatbot, /eventDetail, /newsDetail, /settings, /profile, /userProfile, /facilities/:slug, /professor/:id
  • Campus detail: /campus/:slug → child: campus/:slug/professors

Flows

  • Login → Home → Campus Info → Campus Detail → Staff → Professors → Professor Detail → Book → Appointments.
  • Home → Facilities → pick campus → view gallery → back via breadcrumbs.
  • Search → select campus result → campus detail; or select facility highlight → navigate accordingly.

19) Data Dictionary

Campus

  • name: String — canonical campus name.
  • location: String — city/area detail.
  • imageUrl: String — remote fallback URL.
  • description: String — teaser copy.
  • features: List<String> — bullets.
  • history: String — long text.
  • academicHighlights: List<String>.
  • facilityHighlights: List<String>.
  • photoCaptions: Map<String, String> — assetPath → caption.

ProfessorExtended

  • id, name, title, department, photoUrl, office, bio — descriptive fields.
  • availableSlots: List<String> — times like "Tue 11:00".
  • skills: List<String> — competencies/subjects.
  • achievements: List<String> — accolades/impact.

Appointment

  • id: String — local unique id.
  • userId: String — Firebase uid or demo.
  • professorId: String.
  • start, end: DateTime.
  • location: String.

SearchResultItem

  • section: enum campuses|facilities|professors (future).
  • title: String; subtitle: String.
  • slug: String — for campus routing.

20) Error Handling & UX States

  • Empty: show friendly hints and navigation options.
  • Loading: progress indicators on booking, image fallback on assets.
  • Error: snackbars for booking failures, placeholders for image load errors.
  • Retry: refresh actions on lists and galleries.

21) Authentication Flows

  • Email/password login with validations; Google Sign-In via google_sign_in.
  • Redirects: /login → /home on success; signup similarly.
  • Failure modes: invalid creds, network errors → snackbars and safe states.

22) Booking Lifecycle

  • Initiate from Professor Detail → select slot.
  • Login prompt if unauthenticated; confirmation dialog.
  • Persist JSON in SharedPreferences → auto-navigate to Appointments.
  • Cancel individual or clear-all; reload list on changes.
  • Proposed conflict rules: one active booking per professor per time slot (future backend enforcement).

26) Performance Plan

  • Lazy build lists; avoid excessive rebuilds by using const and keys.
  • Image strategies: asset-first, network fallback, sized containers.
  • Caching: consider cached_network_image in future.

27) Testing Strategy

  • Unit: slugify functions, search scoring.
  • Widget: professors grouped list renders per campus; booking button behaviors.
  • Integration: login → booking → appointments flow.
  • Coverage goal: ≥ 60% initially; expand to ≥ 80% on core components.

28) DevOps & CI Suggestions

  • Lint/format with flutter_lints; check on CI.
  • Run unit/widget tests; build APK on main branch.
  • Artifacts: HTML report and PDF exported on release.

29) Security & Privacy

  • Auth scopes limited to basic profile and email.
  • PII: display name and photo stored locally; future backend storage must follow privacy policies.
  • Threat model: unauthorized access, token leakage; mitigate via Firebase Auth rules and secure storage.

30) Analytics & Telemetry Plan

  • Core events: login success/failure, search queries, booking create/cancel, gallery views.
  • Properties: campus name, professor id (non-PII); avoid storing sensitive data.
  • Dashboards: usage trends, bookings per campus.

31) Release Management

  • Environments: dev (emulator), staging (test devices), prod (store distribution).
